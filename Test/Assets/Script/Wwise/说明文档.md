# Unity → Wwise 自定义音乐播放系统 · 说明文档

> 本文档说明这个测试项目实现了什么、每个脚本的职责与实现方式、依赖与 Wwise 工程配置，
> 以及把它接入**实际游戏项目**时需要程序员处理的事项。

---

## 1. 这个系统做了什么

让 Unity 在**运行时**从磁盘加载任意 wav/mp3/ogg 音频，**不经过 Unity 自身发声**，而是把解码后的 PCM
通过 **Wwise Audio Input 插件**实时喂给 Wwise，由 Wwise 的总线（Main Audio Bus）处理后输出。

核心特性：
- **运行时加载外部音频**（不需要预先导入到工程资源里）。
- **流式 + 环形缓冲**：内存占用恒定（约 1 秒缓冲），与歌曲长度无关。
- **格式支持**：wav（自解析）、mp3（NLayer 解码）、ogg（UnityWebRequest）。
- **文件夹曲库 + Dropdown 选曲 + 播放/停止按钮 + 切歌**。
- **真·交叉淡变**：切歌时旧曲淡出、新曲淡入**同时进行**（两首重叠），用 Wwise 里 Play/Stop 事件各自授权的淡入淡出曲线。
- **转码内存实时监控**。

## 2. 整体架构 / 数据流

```
用户路径/文件夹 → 选曲
      │
      ▼
WwiseAudioInputPlayer（管理层：曲库/Dropdown/切歌/交叉淡变调度）
      │ 每首歌 = 一条 MusicStream（独立 GameObject + 缓冲 + playingID）
      ▼
MusicStream（数据流）
  按扩展名分流解码 → float PCM
      │ 生产者(主线程 Update)：写入环形缓冲
      ▼
  [ 环形缓冲 ringBuffer  约1秒，循环复用 ]
      │ 消费者(Wwise 音频线程 回调)：按声道去交错读出
      ▼
  PostAudioInputEvent(Play_CustomMusic)  ← 各流挂在自己的 GameObject 上
      ▼
Wwise: Audio Input 源 → CustomMusic(Sound) → Main Audio Bus → 输出🔊
```

- **生产者**：`Update()` 里把文件/解码数据增量写进环形缓冲（主线程，绝不在音频线程读磁盘）。
- **消费者**：Wwise 音频线程调 `AudioSamplesCallback` 按声道拉数据。
- 中间环形缓冲解耦两条线程，内存恒定。
- **交叉淡变 = 临时双流**：平时只有一条流（`currentStream`）。切歌时旧流 `Post(Stop)` 开始淡出、新流 `Post(Play)` 开始淡入，**两条流同时喂数据**形成重叠；旧流淡出结束后（等 `fadeOutSeconds`）回收，回到单流。每条流挂在**各自独立的 GameObject** 上，才能让 Stop 事件只命中旧流、不误停新流（`AkAudioInputManager` 按 playingID 分发回调，天然支持多路同时输入）。

## 3. 各脚本职责与实现

> **分层（重要）**：系统拆成两层——
> - **`WwiseAudioInputPlayer`（纯播放引擎，可复用移植）**：接收路径直接读、解码、双流交叉淡变、环形缓冲。不认识文件夹/Dropdown。
> - **`WwiseMusicLibrary`（曲库数据管理，临时/可丢弃/交给程序）**：扫描文件夹、Dropdown、选曲，最终只调 `player.Play(完整路径)`。
>
> 正式项目里曲库层大概率被替换；播放引擎保持不动。两层的唯一契约：**拿到一条能读到文件的完整路径 → `player.Play(path)`**。

### 3.1 `WwiseAudioInputPlayer.cs` —— 纯播放引擎（最重要，可复用）
负责：接收路径 → 加载/解码 → 流式喂 Wwise → 双流交叉淡变 → 内存统计。**不含**曲库/Dropdown/UI。

**对外接口（曲库层 / 游戏逻辑调用这些）：**
| 成员 | 作用 |
|------|------|
| `Play(string path)` | 加载并播放指定文件（正在播则与其交叉淡变）|
| `Stop()` | 带淡出停止当前曲目 |
| `IsPlaying` | 是否有前台曲目在播放 |
| `TranscodeMemoryBytes` | 当前所有存活流转码缓冲占用字节 |

**Inspector 配置项：**
- `audioInputEvent` = Wwise 的 `Play_CustomMusic`（带淡入）
- `stopEvent` = Wwise 的 `Stop_CustomMusic`（带淡出）
- `ringBufferSeconds` = 环形缓冲时长（默认 1 秒）
- `fadeOutSeconds` = **旧流淡出后多久回收资源**，**必须 ≥ Wwise 里 Stop 的淡出时长**（否则尾巴被硬切）。注意：这是回收延时，**不阻塞新曲**——新曲在切歌瞬间就已起播淡入。
- `testAbsoluteFilePath` = 测试专用路径（右键脚本菜单 Debug Play 可试播）

**实现要点：**
- **三种数据源**（`SourceMode`）：
  - `WavStream`：自己解析 WAV 头（fmt/data 块），按需从 `FileStream` 增量读字节，`ConvertSample` 转 float（支持 8/16/24/32-bit int + 32-bit float）。**真流式，最省内存。** 头部用 `ReadFull` 循环读满（`FileStream.Read` 不保证一次读够），读不满即判非法。
  - `Mp3Stream`：`NLayer.MpegFile.ReadSamples()` 按需拉取 float。**真流式。**
  - `MemoryArray`：ogg 等经 `UnityWebRequest` 整段解码进内存再喂（非真流式）。
- **环形缓冲**：`ringBuffer` 交错存 LRLR，`ringReadFrame`/`ringWriteFrame`/`ringAvailableFrames` 三量配 `% 容量` 绕回；读写都在 `lock(ringLock)` 内。
- **去交错**：Wwise 按声道（`channelIndex`）分开拉；`channelIndex==0` 时快照本批帧数 `servedFramesThisBlock`，保证左右读同一批；只在最后声道推进读游标。
- **结束判定**：源读完(`sourceExhausted`) 且缓冲排空 → 回调返回 false，Wwise 停事件。
- **线程安全**：`active`/`sourceExhausted` 用 `volatile`；缓冲访问全程加锁；磁盘 IO 只在主线程。`Dispose` 置空 `ringBuffer` 也在 `lock(ringLock)` 内，且 `AudioSamplesCallback` 进锁先判 `ringBuffer==null`（已回收则填静音返回 false）—— 避免「`StopPlayingID` 到 `AK_EndOfEvent` 注销回调之间音频线程还会回调一两次」时读到已释放缓冲而崩溃（NRE）。
- **MusicStream 抽象**：单首歌的全部状态（解码源 + 环形缓冲 + 游标 + 独立 GameObject + playingID）封装在内嵌类 `MusicStream` 里。播放器只持有 `currentStream`（前台）和 `fadingStreams`（淡出回收中），切歌即「退旧流（`PostStop`）+ 起新流（`Start`）」，`Update()` 给所有存活流补缓冲。
- **交叉淡变流程**：`StartNewStream` → 先 `RetireCurrent`（旧流 `Post(Stop)` 淡出，进 `fadingStreams`，起协程 `DisposeAfter(fadeOutSeconds)` 回收）→ 再起新流 `Post(Play)` 淡入。前台曲自然播完时 `Update` 检测到 `active==false` 自动回收，按钮回到 Play。

### 3.2 `WwiseMusicLibrary.cs` —— 曲库数据管理（临时·可丢弃·交给程序）
负责：扫描文件夹、维护完整路径列表、填充 Dropdown、响应选曲，最终把**完整路径**交给引擎 `player.Play(path)`。
不含任何解码/播放逻辑。

**对外接口：**
| 成员 | 作用 |
|------|------|
| `RefreshLibrary()` | 重新扫描文件夹刷新列表 |
| `PlaySelected()` | 播放 Dropdown 当前选中曲（序号→完整路径→引擎）|
| `TogglePlayStop()` | 播放/停止切换（按钮用）|
| `IsPlaying` | 转发引擎播放状态 |

**Inspector 配置项：**
- `player` = 引用的 `WwiseAudioInputPlayer`
- `folderInputField`（TMP InputField）= 路径输入框（留空用默认）
- `musicFolderPath` = 默认文件夹
- `extensions` = 扫描的扩展名
- `uguiDropdown` / `tmpDropdown` = 选曲下拉框（二选一）

> **序号 vs 路径**：Dropdown 显示文件名、内部传**序号(int)**；`musicFiles`(List\<string\>) 存与序号一一对应的**完整路径**；选中后用序号取路径调 `player.Play(完整路径)`。歌曲的"身份证"= 完整路径。

### 3.3 `WwisePlayStopBotton.cs` —— 播放/停止按钮（极简）
挂在 UI Button 上，点击调用 `library.TogglePlayStop()`，并按 `library.IsPlaying` 切换按钮文字（Play/Stop）。
「播放/停止切换」需知道选了哪首，属曲库层职责，所以引用 **`library`**（不是引擎）。

### 3.4 `WwiseTranscodeMonitor.cs` —— 转码内存监控
定时（默认 0.5s）把以下三项显示到 UI 文本：
- `Transcode`：引擎所有存活流的转码缓冲字节（= 我们音频占用，恒定且小；切歌重叠时约 2 倍）
- `Managed`：C# 托管堆（`Profiler.GetMonoUsedSizeLong`）
- `Reserved`：Unity 总预留（`Profiler.GetTotalReservedMemoryLong`）

引用的是引擎 `WwiseAudioInputPlayer`（`TranscodeMemoryBytes` 在引擎上）。

### 3.5 `WwiseAmbToggle.cs` —— 环境音开关（独立，与上面无关）
挂在 Button 上，点击在 `Play_Amb` / `Stop_Amb` 间切换。原项目已有；已把弃用的 `AkSoundEngine` 改为 `AkUnitySoundEngine`。

### 3.6 `WwiseParameterManager.cs` —— 滑条绑 RTPC（独立）
把若干 Unity Slider 绑定到 Wwise Game Parameter (RTPC)，拖动实时 `SetRTPCValue`。原项目已有；已把弃用的 `AkSoundEngine` 改为 `AkUnitySoundEngine`。

### 3.7 脚本间数据流（谁收什么 / 出什么）

每个脚本的「输入 → 输出」一览。**贯穿全链路的关键数据只有一个：完整文件路径(string)**；它从曲库层流向引擎，引擎再把它变成 float PCM 交给 Wwise。

| 脚本 | 收到什么（输入） | 交给谁 / 输出什么 | 怎么传 |
|------|----------------|------------------|--------|
| **WwisePlayStopBotton** | Unity Button 点击事件 | → `library.TogglePlayStop()`；读 `library.IsPlaying`(bool) 切按钮文字 | 方法调用 / 属性读取 |
| **WwiseMusicLibrary**（曲库层） | 文件夹路径(string，输入框)、选曲序号(int，Dropdown)、按钮调用 | → 引擎 `player.Play(完整路径 string)` / `player.Stop()`；读 `player.IsPlaying` | 方法调用；内部 `musicFiles:List<string>` 序号→路径 |
| **WwiseAudioInputPlayer**（引擎层） | 完整路径(string) via `Play()`；Inspector 的 Wwise 事件/缓冲参数 | → Wwise：注册回调 + `Post(Play/Stop 事件)` + `float PCM`；对外暴露 `IsPlaying`(bool)、`TranscodeMemoryBytes`(long) | 见下「引擎↔Wwise」 |
| **WwiseTranscodeMonitor** | 引擎引用；定时 tick | 读 `player.TranscodeMemoryBytes`(long) → 格式化字符串 → UI 文本 | 属性读取 |
| **WwiseAmbToggle**（独立） | Button 点击；事件名(string) | → `AkUnitySoundEngine.PostEvent(名, gameObject)` | 直接调 Wwise API |
| **WwiseParameterManager**（独立） | Slider 值(float)；RTPC 名(string) | → `AkUnitySoundEngine.SetRTPCValue(名, 值)` | 直接调 Wwise API |
| **AkAudioInputManager**（Wwise集成·幕后） | `PostAudioInputEvent(事件ID, GO, 采样回调, 格式回调)` | 返回 `playingID`(uint)；按 playingID 把音频线程回调路由到对应流 | 委托字典 |

**关系箭头（实线=方法调用/引用，虚线=回调/事件）：**

```
   [Button] ──TogglePlayStop()──▶ [WwiseMusicLibrary] ──Play(完整路径)──▶ [WwiseAudioInputPlayer]
                                         ▲                                        │
                                   IsPlaying(bool)                                │ Post(事件)+注册回调
                                                                                  ▼
   [TranscodeMonitor] ──读 TranscodeMemoryBytes(long)──▶ [WwiseAudioInputPlayer]  [AkAudioInputManager]
                                                                                  │ playingID + 回调路由
                                                                                  ▼
                                                                              [ Wwise 引擎 ] →🔊
   [AmbToggle] ─────PostEvent(名)────▶ [ Wwise 引擎 ]      （独立，不经上面任何脚本）
   [ParameterManager] ─SetRTPCValue(名,值)▶ [ Wwise 引擎 ] （独立）
```

**引擎 ↔ Wwise 的细节（最容易混的部分）：**

| 数据 | 方向 | 方式 |
|------|------|------|
| Play/Stop 事件 | 引擎 → Wwise | `audioInputEvent.Post(go)` / `stopEvent.Post(go)`（每条流在自己的 GameObject 上）|
| 采样/格式回调委托 | 引擎 → Wwise | `AkAudioInputManager.PostAudioInputEvent(...)` 注册，按 `playingID` 入字典 |
| `playingID`(uint) | Wwise → 引擎 | `PostAudioInputEvent` 返回值，用来精确 Stop/回收 |
| 音频格式(采样率/声道) | 引擎 → Wwise | `AudioFormatCallback` 回调里赋值 |
| `float PCM`(samples[]) | 引擎 → Wwise | `AudioSamplesCallback` 在 **Wwise 音频线程**被回调，从环形缓冲去交错填入 |

> **要点**：脚本之间传的几乎都是「路径(string) / 状态(bool) / 字节数(long)」这类轻量数据，靠**方法调用**完成；真正的大数据 **float PCM 只在引擎内部**（主线程写环形缓冲 → 音频线程读）流动，不跨脚本。曲库层与引擎的唯一契约就是一句 `player.Play(完整路径)`，所以曲库层可随意替换/丢弃。

## 5. Wwise 工程侧配置（已完成）

- `CustomMusic`（Sound）挂 **Wwise Audio Input** 源，输出到 **Main Audio Bus**。
- 事件 `Play_CustomMusic`（带淡入）/ `Stop_CustomMusic`（带淡出）。

  `Suspend Audio During Focus Loss = 关`、`Render During Focus Loss = 开`；配合 Unity `Run In Background = 开`。

## 6. 已知限制（接入实际项目要注意）

1. **交叉淡变 = 临时双流**：平时单流；切歌瞬间旧曲淡出 + 新曲淡入重叠（两条 `MusicStream` 同时喂数据），旧流淡出回收后回到单流。重叠期间内存约为单流的 2 倍（各占一份 ~1 秒缓冲），属正常。
   - 连续快速切歌会同时存在多条 `fadingStreams`，各自独立回收；缓冲很小，无碍，但极端连点会短时叠几路输入。
2. **`fadeOutSeconds` 必须 ≥ Wwise 里 Stop 事件的淡出时长**，否则旧曲淡出未结束就被 `Dispose` 硬停，尾巴被切出杂音。它是「旧流回收延时」，不影响新曲起播。
3. **mp3 桌面端必须靠 NLayer**；Unity 自带 `UnityWebRequest` 在桌面端解 mp3 会返回 null。
4. **WAV** 用的是自写解析器（绕开 Unity 不稳定的内置 WAV 解码），支持常见 PCM/float 位深。
5. **路径**：目前用绝对路径（`D:\...` / `C:\Users\User\Music`）。换机器/打包后这些路径不一定存在。

## 7. 接入【实际游戏项目】时，需要程序员做的事

> 这个项目是**功能验证 Demo**。接入正式项目时按下面处理：

### 7.1 Wwise 资源对接
- 在正式项目的 Wwise 工程里建好 `CustomMusic`(Audio Input 源) + `Play_/Stop_CustomMusic` 事件（带淡入淡出）+ 路由到目标总线，并打进 SoundBank、确保运行时加载该 bank。
- 把对应事件拖到 `WwiseAudioInputPlayer` 的 `audioInputEvent` / `stopEvent`。

### 7.2 音频来源策略（**必须明确**）—— 即「曲库层」要怎么重写
`WwiseMusicLibrary` 是临时的 Demo 取数方式（用户手输文件夹）。正式项目里**替换这一层**，决定音乐文件从哪来：
- 玩家自定义文件夹 → 可沿用文件夹扫描思路，但路径改用配置/存档而非硬编码绝对路径。
- 随包内置 → 放 `StreamingAssets`，用 `Application.streamingAssetsPath` 拼路径（注意安卓上 StreamingAssets 在压缩包里，需特殊读取）。
- 跨平台 → 用 `Application.persistentDataPath` 等，不要写死 `C:\...`。
- **唯一契约**：不管曲库层怎么写，最终拿到一条「能读到文件的完整路径」，调 `player.Play(path)` 即可。**播放引擎 `WwiseAudioInputPlayer` 不用动。**

### 7.3 接入游戏 UI / 逻辑
- 现在的「测试按钮 + Dropdown + 曲库层」是 Demo。正式项目里直接调引擎的 `player.Play(具体路径)` / `player.Stop()`，接到你们真正的 UI 或剧情/关卡逻辑上即可，可以完全不要曲库层。
- 播放列表/选曲/上一首下一首等，都在你们重写的曲库层里实现，引擎只管「给路径就播」。

### 7.4 需要程序员评估/决策的点
- **交叉淡变已实现**（切歌时旧曲淡出 + 新曲淡入重叠，见架构第 2 节）。淡入淡出曲线在 Wwise 的 Play/Stop 事件里调，`fadeOutSeconds` 与 Stop 淡出时长对齐即可。
- **是否需要多路常驻同时播放**（比如 BGM + 环境音轨都走 Audio Input 且一直并行）？当前 `MusicStream` 已支持多路并存，但本播放器只把一条当「前台」、其余当「淡出回收」。要常驻多轨需要让播放器维护多条前台流（每轨一条 `MusicStream`），或每轨一个 player + 各自 Wwise 事件/源。
- **mp3 授权**：确认 NLayer(MIT) 可用；或正式项目统一用 wav/ogg。
- **健壮性**：文件不存在/损坏/格式不支持的处理、加载失败的 UI 反馈（现在只打 Log）。
- **内存池**：长时间运行/多音轨时，按 Wwise Advanced Profiler 的 Peak Used 调 `Wwise Initialization Settings` 里的内存池大小。
- **`ringBufferSeconds` 调优**：弱设备/卡顿环境适当调大；追求低延迟可调小（注意欠载风险）。
- **平台验证**：目标平台（尤其主机/移动端）上 NLayer、文件读取、Audio Input、失焦行为都要单独验证。

### 7.5 性能/内存观测
- 运行时看转码占用：用 `WwiseTranscodeMonitor`（Transcode 一行）。
- 看 Wwise 侧内存：Wwise Authoring 连上后用 **Advanced Profiler → Memory**（Used = 当前用量，Peak Used = 峰值，Total Reserved = 预留池上限）。
- 看每个脚本 CPU：Unity Profiler（CPU 模块 + Deep Profile）。注意 `AudioSamplesCallback` 在 Wwise 音频线程，普通视图看不到，需加 ProfilerMarker。
